# CCTV 카메라 → 중앙 서버 직접 접속 스펙 (CCTV팀 전달용)

작성: 서버 파트 / 2026-07-21 · **완료 확인: 2026-07-23** (카메라 앱이 이 스펙대로
9000에 role=CCTV로 직접 붙는 것 확인됨. 통역 다리는 서버파트가 껐음 — PROTOCOL.md
"카메라 통역" 절 참고)
대상: 온카메라 ArUco 검출 앱(PNO-A9081RG, Wisenet Open Platform) 담당자

## 0. 한 줄 요약

카메라 앱이 지금 web_gui(7000)로 보내는 마커 좌표를, **중앙 서버(TLS 9000)에도
`role=CCTV`로 직접** 보내달라는 요청입니다. 서버 쪽은 처음부터 CCTV 접속을
받도록 만들어져 있어 **서버 코드 수정은 0줄**이고, 카메라가 규약대로 접속만
하면 로봇 측위→경로 재계획→QT 표시 루프가 전부 자동으로 살아납니다.

기존 채널은 그대로 유지합니다 (이번 작업은 "연결 교체"가 아니라 "연결 추가"):

```
카메라 ──① RTSP 영상 ────────────────▶ QT (기존 유지)
      ├──② POS 좌표 (TLS, 신규) ─────▶ 서버 192.168.0.8:9000   ← 이번 작업
      └──③ 캘리 명령·스냅샷 ─────────▶ web_gui 7000/7001 (기존 유지)
```

③(CALIB_K 세션, LDC 체크, .ppm 스냅샷)은 대화형·대용량이라 9000에 싣지
않습니다. 지금처럼 web_gui 채널을 계속 씁니다.

> **완료**: ②는 카메라가 이 문서대로 9000에 직접 붙는 것으로 전환 완료됐습니다.
> web_gui의 CAM_POSE→POS 통역 다리는 서버파트가 껐습니다(`RP_CCTV_BRIDGE=0`).
> 혹시 서버 로그에 CCTV role이 반복 재접속하는 게 보이면 다리가 다시 켜져 있다는
> 뜻이니 서버파트에 알려주세요.

## 1. 연결 정보

| 항목 | 값 |
|---|---|
| 주소 | `192.168.0.8:9000` (서버 RPi, 공유기 LAN) |
| 전송 | TCP 위 **TLS 1.2 이상** (평문 TCP 아님) |
| 서버 인증서 | 자가서명. `Server/certs/server.crt` 파일을 드립니다 |
| 인증서 검증 | 권장: `SSL_CTX_load_verify_locations(ctx, "server.crt", NULL)` + `SSL_VERIFY_PEER`. SAN에 `IP:192.168.0.8` 포함돼 있어 IP 접속 검증 가능. 1차 연동 때는 `SSL_VERIFY_NONE`으로 시작해도 됨 |
| 클라이언트 인증서 | 불필요 (서버가 요구하지 않음) |
| 메시지 프레이밍 | **JSON Lines** — JSON 1개 + 개행(`\n`)이 1메시지. 모든 메시지는 `{"type":..., "seq":..., "payload":{...}}` |
| seq | 연결마다 0부터 증가하는 정수. 로그 추적용 (서버가 검사하진 않음) |

참고 구현: 로봇 RPi가 **완전히 같은 방식**으로 이미 붙어 있습니다.
`Paint_Robot/RaspberryPi/src/NetworkManager.cpp` (OpenSSL, TLS1.2+, server.crt
검증, JSON Lines) — 이 파일을 그대로 참고하시면 됩니다.

## 2. 카메라 → 서버 메시지

### 2.1 HELLO (접속 직후 1회, 필수)

```json
{"type":"HELLO","seq":0,"payload":{"role":"CCTV"}}
```

- 접속 후 **10초 안에** 보내야 합니다 (안 보내면 서버가 끊음).
- 서버 응답: `{"type":"ACK","seq":n,"payload":{"msg":"registered as CCTV"}}`
- 같은 role로 재접속하면 서버가 기존 연결을 끊고 새 연결로 교체합니다
  (재부팅/재접속 대응 — 별도 처리 불필요, 그냥 다시 붙으면 됨).

### 2.2 POS (마커 검출 좌표, 15~30Hz)

```json
{"type":"POS","seq":123,"payload":{"corners":[[u0,v0],[u1,v1],[u2,v2],[u3,v3]]}}
```

- **원본 픽셀 좌표 그대로** 보냅니다. undistort(왜곡 보정)·호모그래피 변환을
  카메라에서 하지 말 것 — 좌표계 규약 v0.3에 따라 **변환은 전부 서버가**
  합니다 (undistort → H_marker). 캘리브레이션 데이터는 서버에만 있습니다.
- 지금 CAM_POSE의 `corners:[{"x":..,"y":..},...]` 객체 배열이 아니라
  **`[[u,v],...]` 숫자 배열**입니다. 소수점 유지 권장.

#### ⚠️ 코너 순서 (제일 중요)

서버는 코너 순서로 로봇의 **방향(θ)** 을 계산합니다:

```
순서 = [전좌, 전우, 후우, 후좌]      (로봇 진행방향 기준)

        로봇 정면(진행방향) ▲
            [0]전좌   [1]전우
            [3]후좌   [2]후우
```

- 서버 계산: 중심 = 4점 평균, θ = (후변 중점 → 전변 중점) 벡터 각도.
- ArUco 검출 순서(마커 무늬 기준 좌상→우상→우하→좌하)가 **로봇 정면과 일치하게
  마커가 부착돼 있는지** 확인 필요. 마커의 "위쪽"이 로봇 정면을 향해 있으면
  검출 순서를 그대로 쓰면 되고, 다르면 카메라 앱에서 순서를 재배열해 주세요.
- 순서가 틀리면 로봇 방향이 90°/180° 틀어진 채 계산됩니다. 연동 테스트 때
  로봇을 특정 방향으로 놓고 QT의 POSE θ값과 대조하는 검증을 같이 합시다.

#### MARKER LOST 처리

- **confidence=0(마커 미검출)이면 POS를 보내지 마세요.**
  (서버는 받은 POS를 로봇·QT에 그대로 중계하므로, 빈 좌표를 보내면 그대로
  퍼집니다. heartbeat는 web_gui 채널에서만 유지하면 됩니다.)
- CCTV role에는 무수신 타임아웃이 없습니다 (로봇만 10초 규칙 적용).
  마커가 오래 안 보여도 연결은 유지됩니다.

### 2.3 H_MATRIX (캘리브레이션 결과, 선택)

캘리브레이션 완료 시 결과 번들을 카메라가 직접 올려도 됩니다:

```json
{"type":"H_MATRIX","seq":5,"payload":{"calib":{
  "version":1,
  "K":[[fx,0,cx],[0,fy,cy],[0,0,1]],
  "D":[k1,k2,p1,p2,k3],
  "H_floor":[[..3x3..]],
  "H_marker":[[..3x3..]],
  "marker_height_m":0.25
}}}
```

- 서버가 저장(로그인 사용자에 영속) + QT 중계까지 처리합니다.
- **당장은 안 보내도 됩니다** — 현재는 web_gui가 캘리 완료 시 대신 전송하는
  다리가 이미 구현돼 있습니다. 카메라가 직접 보내게 되면 그 다리를 끄면 됩니다.
- 레거시 `{"H":[[..3x3..]]}` 단일 행렬 형식도 허용됩니다.

## 3. 서버 → 카메라 메시지 (수신 처리)

서버가 CCTV에게 보내는 것이 있으므로 **수신 루프도 필요**합니다:

| type | 내용 | 처리 |
|---|---|---|
| `ACK` | 등록 확인 | 무시 가능 |
| `CMD` `{"cmd":"CALIB_START"}` | QT/관리자가 캘리 시작 버튼을 누름 | 캘리 모드 진입 (web_gui 채널의 CALIB_START와 동일 동작). 당장 구현이 어려우면 무시해도 측위에는 지장 없음 |
| 그 외 | 향후 확장 | **모르는 type은 조용히 무시** (에러로 끊지 말 것) |

## 4. 서버가 POS로 하는 일 (참고)

카메라가 POS만 넣어주면 서버가 자동으로:

1. QT에 원본 그대로 중계 (모니터링용. 로봇에는 원본 픽셀을 보내지 않음 —
   로봇은 좌표를 모르며, 위치 보정은 아래 4번처럼 각도로만 전달)
2. undistort → H_marker로 로봇 pose(x, y, θ) 계산 → QT에 `POSE` 전송 (top-view 표시)
3. 계획 경로에서 0.3 m 초과 이탈 시 재계획 PATH를 로봇에 전송
4. 출발 전 정렬(READY/ALIGN/GO) 판정에 실제 각도로 사용

## 5. 연동 테스트 절차

1. 서버 실행 확인 (서버 RPi): `cd ~/Road_Painter_4th/Server && ./server`
2. 손 테스트 — 카메라 개발 PC에서:
   ```bash
   openssl s_client -connect 192.168.0.8:9000 -quiet
   # 붙은 뒤 아래 한 줄 입력(개행 포함):
   {"type":"HELLO","seq":0,"payload":{"role":"CCTV"}}
   # → {"type":"ACK",...,"registered as CCTV"} 가 오면 성공
   {"type":"POS","seq":1,"payload":{"corners":[[100,100],[200,100],[200,200],[100,200]]}}
   ```
3. 서버 로그에서 확인: `[접속] CCTV <ip>` 및 POS 처리 로그.
   캘리브레이션이 아직 없으면 `POS 수신했으나 캘리브레이션 없음 - 중계만 함`이
   뜨는 게 정상입니다 (H_MATRIX 들어오면 pose 계산 시작).
4. 관리자 창(`http://192.168.0.8:8081/logs`)에서 `[tap] CCTV->SRV POS ...`가
   흐르면 전 구간 OK.
5. 방향 검증: 로봇(마커)을 x축 방향으로 놓고 QT/관리자창의 POSE θ≈0° 확인,
   90° 돌려놓고 θ≈90° 확인 (코너 순서 검증).
