## 1. 개요 (Introduction)

### 1.1 목적 (Purpose)

본 문서는 '인프라 비전 측위 기반 자율 구획 도장 로봇 시스템(Road-Painter)' 개발을 위한 시스템 사양 및 요구사항을 명시합니다. 고정형 CCTV를 인프라 측위 센서로 변환하여 GNSS 음영 구역 내 저비용·고정밀 주행 및 도포 제어를 구현하기 위해, 조원 전체가 준수해야 할 정량적 기술 규격을 정의하는 것을 목적으로 합니다.

### 1.2 측위·제어 기본 원칙

본 시스템의 구동계는 바퀴 엔코더를 사용하지 않고 **스텝모터**를 사용합니다. 측위·제어 계층의 역할은 다음과 같이 정의합니다.

- **CCTV(고정 오버헤드 카메라) = 절대 측위 센서**: 카메라에 설치한 Open Platform 앱이 ArUco 마커를 직접 검출(베이스라인)하여 로봇의 절대 위치·헤딩의 원천을 제공하며, 스텝모터 오픈루프 구동의 누적 오차(슬립, 탈조, 바퀴 지름 오차)를 보정하는 유일한 절대 기준입니다.
- **스텝모터 스텝 카운트 = 고속 상대 측위(데드레커닝)**: 스텝모터는 명령 스텝 수와 실제 회전량이 (탈조가 없는 한) 1:1로 대응하므로, STM32가 발행한 스텝 카운트를 고속 상대 위치 추정에 사용합니다. 별도 엔코더 없이 스텝 카운트가 가상 엔코더 역할을 수행합니다.
- **폐루프의 구성**: 바퀴 축 수준의 속도 폐루프는 존재하지 않습니다. 대신 "카메라 앱 ArUco 검출 → 서버 좌표 변환(POSE 생성) → 로봇 RPi 경로 추종 루프(10~15Hz) → 스텝 속도" 형태의 외곽 폐루프가 경로 추종 오차를 상시 보정합니다.
- **PATH 1회 + POSE 상시 원칙**: 경로(PATH)는 작업 시작 시 1회 로봇에 전달되지만, 주행 중에는 POSE(절대 측위) 피드가 상시 공급되어야 폐루프가 성립합니다. **POSE 없이 PATH만으로 주행하는 것은 오픈루프이므로 금지** — POSE 상실 시 로봇은 즉시 정지합니다.
- **탈조(step loss) 리스크 관리**: 토크 마진 확보·사다리꼴 가감속 프로파일·CCTV 측위와 스텝 카운트 추정치의 상시 교차 검증(괴리 임계 초과 시 탈조 판정)을 요구사항으로 규정합니다.

### 1.3 좌표 변환 기본 원칙 (2평면 호모그래피)

호모그래피는 평면↔평면 변환이므로, 시스템은 용도가 다른 **2종의 호모그래피를 엄격히 구분**하여 사용합니다. 두 변환의 혼용은 마커 높이 시차(parallax)로 인해 수십~수백 mm의 계통 오차를 유발하므로 금지합니다.

| 구분 | 대상 평면 | 용도 | 캘리브레이션 방법 |
|---|---|---|---|
| H_marker (마커 평면) | 바닥 + 마커 장착 높이 h | 로봇(마커) 측위 → 지면 (X,Y) | 바닥 평면을 실측 높이 h만큼 법선 오프셋한 평면에서 **해석적 유도** + 로봇 스팟 체크 검증 (부록 B Step 3~4) |
| H_floor (바닥 평면) | 바닥 (높이 0) | 사용자 작도 경로의 좌표화, 도포 결과 검사 | 소형 보드 3지점 **PnP**로 카메라 자세·바닥 평면 확정 후 해석적 유도 (부록 B Step 2~3) |

- H_marker는 마커 높이 평면 기준으로 산출되므로 **로봇 측위에 한해 높이 시차가 원천 상쇄**됩니다. 단, 이 상쇄는 로봇 측위에만 유효하며 바닥 도형 좌표화에는 반드시 H_floor를 사용해야 합니다.
- 캘리브레이션의 기준 좌표(ground truth)는 **체커보드 격자(칸 크기 × 인덱스) 등 물리 실측 기반 값**을 사용해야 하며, **오도메트리(스텝 카운트) 주행 결과를 기준 좌표로 사용하는 것을 금지**합니다. 오픈루프 오도메트리 오차(거리 1~5%, 헤딩 드리프트)가 캘리브레이션에 내장되는 순환 참조가 발생하기 때문입니다. 오도메트리 주행 비교는 검증(validation) 용도로만 사용합니다(SRN-004).
- 상쇄가 깨지는 조건(바닥 요철, 로봇 기울어짐에 의한 h 변동)을 관리하기 위해, 기구 설계 시 **마커 장착 높이 h를 최소화**해야 합니다. 로봇 기울어짐 θ에 의한 수평 오차는 근사적으로 h·sinθ이며(h=200mm, θ=1°일 때 약 3.5mm), h 최소화는 시차 잔차와 기울어짐 민감도를 동시에 저감합니다.

### 1.4 시스템 기술 스택 및 환경

- 개발 언어: C++ (Qt, STL), Embedded C (STM32)
- 운영체제 및 환경: 서버·로봇 — Linux (Raspberry Pi 4) / 관제 클라이언트 — Windows 노트북 (Qt Creator), 전 구간 OpenSSL
- 비전 라이브러리: OpenCV
- 인터페이스 규격: RTSP/IP, LAN/Wi-Fi (TCP/TLS), JSON, UART

### 1.5 정밀도 스펙 정의

정밀도 목표는 다음 조건부로 정의합니다: **"카메라 시야 중앙 존(예: 3×3m), 주행 속도 3~5cm/s, 직선·직각 경로에서 최대 이탈 ±5mm"**. 영상 가장자리 및 저설치각 영역에서는 본 스펙을 보장하지 않습니다.

## 2. 시스템 아키텍처 및 통신 사양 (System Architecture & Interfaces)

시스템 인터페이스 제어(ICD)의 간소화를 위해 통신 계층의 역할을 명확히 분리하고 규격화합니다.

### 2.1 네트워크 토폴로지

시스템의 모든 노드는 **단일 Wi-Fi 공유기(라우터)가 구성하는 하나의 로컬 네트워크(LAN)**에 속합니다.

| 노드 | 접속 방식 | 비고 |
|---|---|---|
| 한화비전 PNO-A9081RG (4K AI 불릿) | 공유기 유선 LAN 포트 | 급전은 PoE 인젝터 경유(공유기가 PoE 미지원인 경우) |
| 서버 RPi | 공유기 유선 LAN 포트 (권장) | 좌표 변환·경로 계산·캘리브레이션 주체 |
| 사용자 노트북 (Windows, Qt) | 유선 또는 Wi-Fi | 관제 UI 전용 |
| 로봇 RPi | Wi-Fi | 이동체, **경로 추종 루프 수행** |

- 카메라→공유기→서버 경로는 유선이므로 RTSP 지연·손실이 작습니다. 무선 구간은 로봇(필수)과 노트북(선택)에 한정합니다.
- 모든 노드는 고정 IP 또는 DHCP 예약으로 주소를 고정합니다.
- 데모 환경에서는 외부 간섭을 피하기 위해 전용 공유기 사용을 권장합니다.

### 2.2 물리적 및 프로토콜 인터페이스 정의

- **IF-RTSP-001 (카메라 ↔ 서버 RPi, 공유기 유선 LAN 경유)**: 서버 RPi는 공유기 LAN을 통해 한화비전 PNO-A9081RG로부터 RTSP 스트리밍을 수신합니다. 메인 스트림은 관제·결과 검사용이며, 서브 스트림은 **서버 폴백 검출 전용**입니다(베이스라인 측위는 IF-TCP-003의 온카메라 검출).
- **IF-TCP-003 (카메라 앱 → 서버 RPi, 측위 피드)**: 카메라에 설치된 Open Platform 앱이 원시 프레임에서 검출한 **ArUco 코너 픽셀 좌표 + 프레임 타임스탬프 + 신뢰도**를 TCP 경량 메시지로 서버에 15~30Hz 송신합니다. 본 시스템 측위의 **베이스라인 입력**입니다.
- **IF-HTTP-002 (서버 RPi → 카메라, SUNAPI)**: 서버는 카메라의 HTTP API(SUNAPI)로 프로파일 설정 확인·변경, 스냅샷 요청, 온카메라 앱의 기동·상태 확인을 수행합니다. 실시간 제어 계층에는 사용하지 않습니다.
- **IF-WIFI-001 (서버 RPi ↔ 로봇 RPi)**: 고정 서버와 이동 로봇 간 무선 링크는 Wi-Fi 환경에서 TCP/TLS 기반 OpenSSL 보안 세션을 적용합니다. 메시지 규격은 부록 C(ICD)를 따릅니다.
- **IF-NET-002 (서버 RPi ↔ 사용자 Qt 클라이언트, Windows 노트북)**: 서버와 관제 클라이언트 간 링크(유선 또는 Wi-Fi)는 TCP/TLS 기반 OpenSSL 보안 세션을 적용합니다.
- **IF-UART-001 (로봇 RPi ↔ STM32)**: 로봇 탑재 RPi 브릿지와 메인 제어기 STM32 간 제어 명령(좌/우 스텝모터 목표 스텝 속도, 도장 액추에이터 명령) 및 상태 피드백(누적 스텝 카운트, 상태 플래그) 전송을 위한 시리얼 UART 유선 인터페이스를 구성합니다. 프레임 구조는 부록 C.5를 따릅니다.

### 2.3 카메라 스트림 프로파일 (메인/서브)

IP 카메라는 하나의 센서 영상을 **서로 다른 인코딩 설정의 여러 스트림(프로파일)으로 동시에 송출**할 수 있으며, 프로파일마다 별도의 RTSP URL이 존재합니다. 본 시스템은 카메라 1대에서 **총 2개 스트림(메인 1 + 서브 1)**을 수신합니다. ("서브 스트림 2개"가 아님에 유의.)

| 항목 | 메인 스트림 (관제·검사용) | 서브 스트림 (폴백 검출용) |
|---|---|---|
| 용도 | Qt 대시보드 표시, 결과 검사 캡처 | 온카메라 앱 장애 시 서버 ArUco 검출 |
| 해상도 | 4K (3840×2160) | 검출에 충분한 수준 (예: 1080p) |
| GOP | 카메라 기본값 | 짧게 (예: fps와 동일 = 1초 간격) |
| B-프레임 | 허용 | **비활성** |
| 3D-DNR | 허용 | **비활성** |
| 서버 수신 버퍼 | 일반 버퍼링 (무끊김 우선) | 최신 1장만 유지 (지연 최소) |

프로파일 설정 근거 (용어는 부록 D 참조):
- **GOP**: I-프레임(독립 복호 가능 프레임) 간 간격. 짧을수록 디코더가 빨리 동기화되어 지연·손상 복구에 유리합니다(비트레이트는 다소 증가).
- **B-프레임**: 미래 프레임을 참조하여 압축하는 프레임 유형. 압축률은 높지만 참조 대기만큼 지연이 추가되므로 측위 스트림에서는 금지합니다.
- **3D-DNR**: 여러 프레임을 시간축으로 평균하여 노이즈를 줄이는 카메라 기능. 정지 장면은 깨끗해지지만 **움직이는 마커에 잔상(고스팅)**을 만들어 서브픽셀 코너 검출을 훼손하므로 측위 스트림에서는 반드시 끕니다. 별도 구현이 아니라 **카메라 웹 설정 메뉴에서 프로파일별로 OFF**하는 설정 작업입니다.

카메라 사양 참고: PNO-A9081RG — 4K(3840×2160@30fps), 1/1.8″ CMOS, 4.5~10mm **전동 바리포컬** 렌즈(2.2x), H.265/H.264/MJPEG, 멀티 프로파일 지원. 4K로 폭 3m 작업 영역을 관측하면 약 0.8mm/px의 화소 밀도가 확보됩니다. 전동 바리포컬이므로 **캘리브레이션 완료 후 줌/포커스 조작을 금지**합니다(조작 시 부록 B Step 1부터 재수행).

### 2.4 제어 루프 계층 구조

| 루프/피드 | 위치 | 주기 | 역할 |
|---|---|---|---|
| 측위 피드 | 카메라 앱 → 서버 → 로봇 | 10~15Hz | ArUco 검출(카메라) → H_marker 변환·POSE 생성(서버) → 로봇 전달 |
| 경로 추종 루프 | **로봇 RPi** | 10~15Hz | PATH + POSE → 지연 보상 → 횡방향/헤딩 오차 → (v, ω) → 좌/우 sps |
| 스텝 펄스 생성 루프 | STM32 (타이머/하드웨어) | 스텝 주파수 단위 | 좌/우 스텝모터 펄스 발생, 가감속 프로파일 실행 |
| 안전 워치독 | 로봇 RPi / STM32 | 상시 | POSE/명령 수신 타임아웃 시 정지 + 노즐 차단 |

### 2.5 베이스라인 아키텍처 — 온카메라 ArUco 검출 (Wisenet Open Platform)

PNO-A9081RG는 Wisenet Open Platform을 지원하며(SDK: STEP 포털, 계정 확보됨), 본 프로젝트는 **온카메라 검출을 베이스라인으로 채택**합니다.

- **베이스라인**: 카메라에 설치한 Open Platform 앱(.cap)이 원시 프레임에서 ArUco를 검출하고, **코너 픽셀 좌표 + 프레임 타임스탬프 + 신뢰도만** 서버에 송신(IF-TCP-003). 서버는 H_marker 변환으로 펜 끝 POSE를 만들어 로봇에 중계하고, **경로 추종 루프는 로봇 RPi가 수행**합니다.
- **채택 근거 (부하 최소화)**: ① 서버 RPi의 측위용 영상 디코딩 부하 제거, ② 측위 경로에서 인코딩·전송·디코딩 단계가 빠져 지연 대폭 단축, ③ 측위용 대역폭이 영상에서 좌표 몇십 바이트로 감소, ④ 서버는 고주기 제어 대신 좌표 변환·중계만 수행, ⑤ "인프라 카메라가 로봇의 눈" 가치 제안 강화.
- **폴백 (필수 구현)**: 서버의 측위 입력을 "측위 피드(코너 px, 타임스탬프, 신뢰도)" 인터페이스로 추상화하여, 온카메라 앱 장애·개발 지연 시 **서버가 서브 스트림에서 자체 검출하는 경로로 런타임 자동 전환**합니다. 카메라 앱 완성 전까지는 폴백 경로로 통합 시험을 진행할 수 있습니다.
- **제약**: WiseAI는 사람·차량 등 고정 객체용 내장 AI로 ArUco 검출에는 직접 사용할 수 없습니다(커스텀 앱 필요). 카메라 SoC 리소스 한계 내에서 검출 해상도·주기를 조정하며, 앱은 프레임 타임스탬프를 반드시 포함해야 합니다.

## 3. 기능 요구사항 명세 (Functional Requirements)

각 시스템 블록이 수행해야 하는 기능적 요구조건을 식별 부호(SR) 체계로 구성합니다.

### 3.0 온카메라 검출 앱 (CCTV / Wisenet Open Platform)

**SR-CAM-001 (ArUco 검출 및 측위 피드 송신)**
- 카메라 내 Open Platform 앱은 원시 프레임에서 로봇 상판 ArUco 마커를 검출(서브픽셀 코너 정밀화 포함)하고, **(코너 픽셀 4점, 프레임 타임스탬프, 검출 신뢰도)**를 IF-TCP-003으로 서버에 15~30Hz 송신해야 합니다.
- 검출 실패 프레임은 신뢰도 0으로 보고하며, 앱은 SUNAPI 또는 헬스 메시지로 동작 상태를 확인 가능해야 합니다.
- 카메라 SoC 부하를 고려하여 검출 입력 해상도·ROI·주기를 설정 가능해야 합니다.

**SR-CAM-002 (폴백 지원)**
- 앱 장애·미탑재 시 서버가 서브 스트림 기반 자체 검출로 전환할 수 있도록, 측위 피드 메시지 형식은 서버 내부 검출 출력과 동일해야 합니다(2.5절).

### 3.1 비전 서버 및 영상 처리 모듈 (Server RPi / Vision)

**SR-VIS-001 (채널 수신)**
- GStreamer 파이프라인과 OpenCV를 연동하여 카메라의 RTSP(메인 스트림)를 누락 없이 수신해야 합니다(관제·검사용, 일반 버퍼링).
- **폴백 모드에서만**: 서브 스트림 수신부를 활성화하며, 전용 스레드가 항상 최신 프레임 1장만 유지하고 구프레임을 즉시 폐기해야 합니다(최신성 우선).

**SR-VIS-002 (영상 전처리 파이프라인 — 1차/2차 분담)**
- **1차 전처리(카메라 내장)**: 디노이즈, 밝기·대비 보정, 선명도 향상을 카메라 프로파일 설정으로 수행하여 서버(RPi4) 연산 부하를 저감하고 인코딩 전 원신호 품질을 확보합니다.
  - 단, **측위용 서브 스트림 프로파일에서는 시간축 노이즈 제거(3D-DNR)를 비활성화**해야 합니다. 이동 중인 마커에 모션 블러·고스팅을 유발하여 서브픽셀 코너 검출 정밀도와 지연 특성을 훼손하기 때문입니다. 선명도(샤프닝)는 과도한 헤일로가 코너 검출에 영향을 주지 않는 수준으로 제한합니다.
- **2차 전처리(서버)**: 호모그래피 연산 전단에 이미지 전처리 루틴을 배치해야 합니다. 렌즈 왜곡 보정을 필수로 하고, 잔여 보정(대비 보정 등)은 측위 지연에 영향을 주지 않는 최소 구성으로 합니다. 가우시안·미디언 필터 기반 디노이즈, 언샤프 마스크 등은 1차 전처리로 부족한 경우에 한해 적용합니다.

**SR-VIS-003 (2평면 호모그래피 좌표 변환)**
- OpenCV로 카메라 렌즈 왜곡을 보정하고, 1.3절에 정의된 **H_marker(마커 평면)와 H_floor(바닥 평면) 2종의 호모그래피를 각각 산출·관리**해야 합니다.
- H_floor 산출: 소형 체커보드를 도장 영역 내 3지점 이상에 잠깐씩 놓고 PnP(`cv::solvePnP`)로 카메라 외부 파라미터와 바닥 평면을 확정한 뒤 해석적으로 유도합니다(부록 B Step 2~3).
- H_marker 산출: 확정된 바닥 평면을 실측한 마커 높이 h만큼 법선 방향으로 평행 이동한 평면에서 해석적으로 유도하고, 로봇 스팟 체크로 검증합니다(부록 B Step 3~4). **오도메트리 주행 결과를 기준 좌표로 사용하지 않습니다.**
- 용도별 적용을 강제합니다: 로봇 측위 → H_marker, 사용자 작도 경로 좌표화·결과 검사 → H_floor. 소스코드 수준에서 두 변환의 오용을 방지하는 인터페이스 분리를 권장합니다.
- (선택 기능) 상시 기준 마커(부록 B Step 7)를 설치한 경우, 매 세션 시작 시 그 검출 좌표를 저장값과 비교하여 임계치 초과 시 재캘리브레이션 경고를 발생시킵니다(카메라 마운트 미세 이동 감지). 미설치 시 "카메라 접촉 금지, 접촉·이동 시 재캘리브레이션" 운영 규칙으로 대체합니다.

**SR-VIS-004 (POSE 생성 및 중계)**
- 측위 피드(카메라 앱 코너 px, 폴백 시 자체 검출)를 수신하여 `undistortPoints` → **H_marker** 변환 → 마커 코너 4점 기하로 헤딩 각 산출 → 펜 끝 오프셋 적용으로 **POSE(펜 끝 x, y, θ, 원본 프레임 타임스탬프, 신뢰도)**를 생성해야 합니다(제어 기준점은 로봇 중심이 아닌 **펜 끝**).
- 직전 위치 대비 물리적으로 불가능한 점프(예: 1프레임 5cm 초과)는 이상치로 폐기하고, 2~3프레임 이동평균 수준의 경량 평활화를 적용합니다.
- 생성된 POSE를 **10~15Hz 정주기로 로봇 RPi에 중계**해야 합니다(부록 C.2). 검출 실패 시 신뢰도 0을 전달하고, 연속 실패가 임계 시간(예: 1초)을 넘으면 정지를 지시해야 합니다.
- 마커는 검출 정밀도를 위해 가능한 한 크게, 수평으로, **가능한 한 낮은 높이에** 장착합니다(1.3절).

**SR-VIS-005 (타임스탬프 체계)**
- 측위 피드의 타임스탬프는 **카메라 프레임 취득 시각**을 기준으로 하며, POSE에 그대로 실어 로봇까지 전달해야 합니다(지연 보상은 로봇 측 SR-MPU-005가 수행).
- 서버·로봇 간 시각 동기화(오프셋 추정) 방법을 정의해야 하며, 총 측위 지연(검출→POSE 도착)은 통합 전 실측하여 수치로 문서화해야 합니다.

### 3.2 관제 클라이언트 모듈 (Qt Client)

**SR-QT-001 (대시보드 GUI 구현)**
- 수신된 영상 스트림 및 로봇의 현재 상태(속도, 헤딩, 스텝모터 구동 상태, 배터리 전압, 탈조 경고 등)를 단일 관제 UI 화면에 모니터링 출력해야 합니다.

**SR-QT-002 (경로 작도 및 벡터 변환)**
- 사용자가 화면에 입력한 마킹 경로(직선, 사각형, 주차 구획선)를 실시간 인식하여 좌표점(Waypoint) 시퀀스로 변환하고, 로봇 구동 명령 패킷으로 구성하여 서버로 송신해야 합니다.
- 화면 픽셀 좌표는 **H_floor(바닥 평면 호모그래피)**로 작업장 좌표로 변환해야 합니다.
- 경로는 **획(stroke) 단위로 분해**하고, 각 획 시작점 전방에 **리드인 구간(30~50cm, 노즐 OFF)** 및 종점 후방에 리드아웃 구간을 자동 부가해야 합니다. 로봇은 리드인 구간에서 횡방향·헤딩 오차를 수렴시킨 후 실제 시작점 **위치 통과 시점**에 도포를 개시합니다.

**SR-QT-003 (결과 검사 및 정량 리포트)**
- 작업 완료 후 메인 스트림 고해상도 프레임에서 도포된 선을 색 기반으로 검출하고 **H_floor**로 좌표화하여, 계획 경로 대비 **평균/최대 이탈(mm) 및 누락 구간**을 정량 산출·표시해야 합니다.

### 3.3 로봇 메인 MPU (Raspberry Pi 4B)

**SR-MPU-001 (네트워크 패킷 중계)**
- Wi-Fi로 수신된 바이너리 제어 버퍼 패킷을 실시간 파싱하여 하위 단에 규정된 프레임 구조로 변환해야 합니다.

**SR-MPU-002 (전용 디바이스 드라이버 구현)**
- 타워램프, 상태 알림용 부저의 정밀 동작 제어를 위해 커널 계층과 사용자 계층을 분리한 임베디드 GPIO 및 PWM 디바이스 드라이버를 직접 설계·빌드해야 합니다.

**SR-MPU-003 (하드웨어 제어 모듈과 통신)**
- 경로 추종 루프(SR-MPU-005)가 산출한 (v, ω)를 기구 상수(바퀴 지름, 축간거리, 기어비, 스텝각)를 이용해 좌/우 스텝모터 목표 스텝 주파수(sps)로 변환하고, STM32에 UART로 끊김 없이 전송해야 합니다.
- STM32로부터 누적 스텝 카운트·상태 플래그를 주기(10Hz) 수신하여 STATUS 메시지로 서버에 상향 중계해야 합니다.
- 서버↔로봇 메시지의 종류·필드·주기 및 UART 프레임 구조는 **부록 C(ICD)**를 따릅니다.

**SR-MPU-005 (경로 추종 루프 — 본 시스템의 유도 제어 주체)**
- 서버로부터 작업 시작 시 **PATH(획·웨이포인트·리드인/리드아웃·노즐 ON/OFF 위치 포함)**를 수신·저장해야 합니다.
- 주행 중 **POSE(10~15Hz)**를 수신하여 지연 보상("POSE 타임스탬프 이후 자신이 발행한 속도 명령의 적분"으로 현재 위치 추정)을 수행한 뒤, 현재 목표 획 선분에 대한 **횡방향 오차와 헤딩 오차**를 계산하고 (v, ω)를 생성해야 합니다(10~15Hz).
- 횡방향 오차 항만 사용하는 제어는 금지하며 헤딩 오차 항을 반드시 병용합니다(출렁임 방지). |오차| 임계 초과 시 감속, 2차 임계 초과 시 정지 후 리드인 방식 재진입.
- 노즐 ON/OFF는 PATH에 정의된 **위치 통과 기준**으로 로봇이 자체 판정하여 STM32에 하향합니다(기구 지연 선행 보정 포함).
- **POSE 300ms 이상 미수신 또는 신뢰도 0 연속 시 즉시 정지 + 노즐 차단**(1.2절 "PATH 1회 + POSE 상시" 원칙).

**SR-MPU-004 (커널 인터럽트 기반 센서 관제 및 안전 제어)**
- 전방 초음파/ToF 센서의 Echo 및 인터럽트 핀을 리눅스 커널 GPIO 인터럽트 서브시스템에 등록하는 전용 드라이버를 작성해야 합니다. 장애물 임계거리 진입 시 유저 공간을 거치지 않고 커널 내부 ISR에서 하위 제어기에 긴급 정지 명령을 하향 송신해야 합니다.

### 3.4 로봇 메인 구동 및 하드웨어 제어 모듈 (STM32 Firmware)

**SR-STM-001 (스텝모터 기반 고정밀 주행 제어)**
- STM32 타이머 인터럽트 하드웨어 자원을 직접 제어하여 좌/우 스텝모터의 **스텝 펄스 주파수를 독립 생성**해야 하며, 직선 주행 및 제자리 회전 기능을 보장해야 합니다.
- 기동·정지·속도 변경 시 **사다리꼴(가감속 램프) 속도 프로파일**을 적용하여 탈조를 방지해야 합니다. 최대 속도·가속도는 부하 조건에서 토크 마진을 확보한 값으로 제한합니다.
- 좌/우 모터의 **누적 스텝 카운트를 상시 유지**하고 상위(MPU)에 주기 보고해야 합니다. 이 값은 서버 측 지연 보상 및 탈조 교차 검증의 입력이 됩니다.
- 회전 중심(좌우 바퀴 축 중앙)에 도장 기구를 장착하는 기구 설계를 전제로, 제자리 회전 시 펜/노즐 위치가 이동하지 않아야 합니다.

**SR-STM-002 (도장 모터 제어)**
- MPU로부터 도장 시작 신호를 받으면 STM32 타이머 기반 PWM 신호로 서보모터(도장 액추에이터)를 제어해야 합니다.
- 도포 ON/OFF는 시간 기준이 아닌 **경로상 위치 통과 기준** 명령을 수행하며, 액추에이터 기구 지연(수십 ms)을 실측하여 트리거를 선행 보정해야 합니다.

**SR-STM-003 (도막 품질 균일화 로직)**
- 주행 속도의 실시간 변화 및 가감속 상태(스텝 주파수 램프 구간 포함)에 연동하여 마킹 노즐/펜 구동부의 PWM 듀티비를 자동 보정함으로써 선 두께·농도 편차를 방지해야 합니다.

**SR-STM-004 (상위 제어 데이터 수신 및 실시간성 보장)**
- MPU로부터의 고빈도 제어 패킷을 데이터 누락·타이밍 지연 없이 수신할 수 있도록 통신 인터페이스 및 수신 버퍼(링 버퍼 + UART 인터럽트/DMA) 구조를 구성해야 합니다.
- 수신·파싱 처리는 스텝 펄스 생성 타이머 루프에 영향을 주지 않도록 하드웨어 자원을 분리하여 백그라운드에서 독립 수행되어야 합니다.

**SR-STM-005 (탈조 감지 대응)**
- 서버가 CCTV 측위값과 스텝 카운트 기반 추정치의 괴리를 감시하여 임계 초과 시 탈조로 판정하고 감속·정지·재진입(리드인 방식)을 지시하면, STM32는 해당 명령을 최우선 수행해야 합니다.

### 3.5 로봇 전원 관리 (전원 분배 보드, 회로)

**SR-PWR-001 (로봇 메인 전원)**
- 로봇 배터리는 공칭 전압 11.1V 이상의 3S 리튬 폴리머(Li-Po) 또는 동등 이상 전원을 공급해야 하며, 전압 강하로 인한 오작동(특히 저전압 시 스텝모터 토크 저하 → 탈조 위험 증가)을 방지하기 위해 실시간 전압 모니터링 회로를 포함해야 합니다.

**SR-PWR-002 (LED 조명)**
- 전원이 정상 공급되면 기구 앞면 LED 스트립으로 사용자에게 시각적으로 알려야 합니다.

**SR-PWR-003 (전원 격리 분배 및 전압 강하)**
- 전원 분배 보드는 **스텝모터 드라이버 구동을 위한 대전류 라인**을 구성하되, 스텝모터의 스위칭 노이즈가 메인 제어부로 인입되지 않도록 디커플링 회로(콘덴서 필터 등) 또는 레귤레이터 경로를 분리해야 합니다.
- DC-DC 벅 컨버터로 5V 정전압을 생성하여 Raspberry Pi, STM32, 도장 서보모터에 안정 분배해야 합니다.

## 4. 비기능 및 신뢰성 요구사항 (Non-Functional Requirements)

**SRN-001 (통신 유실 시 Fail-Safe)**
- 무선 링크(Wi-Fi) 다운 또는 임계 시간(예: 300ms) 이상 패킷 수신 지연 시, 하위 제어기는 즉시 비상 정지(Emergency Stop) 모드로 전환하고 도포 노즐을 강제 차단(Shut-off)해야 합니다. 정지 시에도 스텝모터는 감속 램프를 거쳐 정지하여 탈조·미끄러짐 없이 위치 추정 일관성을 유지해야 합니다.

**SRN-002 (장애물 급제동)**
- 자율 주행 중 전방 초음파/ToF 센서 범위 내 미확인 장애물 유입 시, 탑재 펌웨어 수준에서 최우선 하드웨어 인터럽트를 트리거하여 물리 급제동을 강제 수행해야 합니다.

**SRN-003 (측위 주기 제어)**
- 가상 오도메트리 추정값의 송수신 인터벌은 유도 루프(5~10Hz) 안정성을 해치지 않는 정주기 성능을 유지해야 합니다. 측위 상실(연속 검출 실패)이 임계 시간을 넘으면 정지 + 노즐 차단으로 전환해야 합니다.

**SRN-004 (정밀도 검증 기준)**
- 캘리브레이션 정적 검증: 기지 위치의 기준 마커 변환 좌표 오차 **중앙 존 3mm 이하** (H_marker, H_floor 각각 검증).
- 동적 검증: "2m 직진" 명령 후 비전 측정 실주행 거리와 스텝 카운트 환산 거리를 비교하여 호모그래피 교차 검증 및 스텝-거리 스케일(바퀴 지름 보정값)을 확보해야 합니다. 본 실험은 캘리브레이션이 아닌 **검증 전용**이며, 오도메트리 주행 결과를 캘리브레이션 기준 좌표로 역사용하는 것을 금지합니다(1.3절).
- 최종 인수 기준: 중앙 존, 3~5cm/s, 직선·직각 경로에서 최대 이탈 ±5mm, 동일 도형 3회 반복 재현.

**SRN-005 (보안)**
- 서버↔로봇, 서버↔클라이언트 전 구간은 TCP/TLS(OpenSSL) 보안 세션을 유지해야 하며, 인증되지 않은 피어의 제어 명령은 거부해야 합니다.

**SRN-006 (사용자 인증 및 설정 이력 관리)**
- 서버는 사용자 계정(ID/PW)을 관리하고, Qt 클라이언트는 로그인 후에만 작업 명령을 발행할 수 있어야 합니다(TLS 위 응용 계층 인증).
- 캘리브레이션 산출물(yaml)·작업 로그·결과 리포트는 사용자·세션별로 버전 관리하여 서버에 저장하고, Qt·로봇이 접속 시 필요한 최신 설정을 배포(CONFIG_SYNC)해야 합니다.

## 부록 B. 캘리브레이션·호모그래피 상세 시나리오 (Calibration & Homography SOP)

### B.1 개념 정리

- **카메라 내부 파라미터(intrinsic) 캘리브레이션**: 렌즈 자체의 성질(초점거리, 주점, 왜곡계수)을 구하는 작업. 카메라 행렬 K와 왜곡계수 dist가 산출물이며, 렌즈/줌/포커스 설정이 바뀌지 않는 한 1회로 충분합니다.
- **호모그래피(H)**: 왜곡이 보정된 영상 픽셀 (u, v)를 특정 "평면" 위의 실좌표 (X, Y)로 옮기는 3×3 행렬. `[X', Y', W]ᵀ = H·[u, v, 1]ᵀ, (X, Y) = (X'/W, Y'/W)`. 평면마다 별개의 H가 필요하므로 본 시스템은 H_floor(바닥)와 H_marker(마커 높이) 2개를 사용합니다(1.3절).
- **PnP(Perspective-n-Point)**: 크기를 아는 패턴(체커보드)의 코너들이 영상 어디에 맺혔는지로부터, 카메라와 패턴 사이의 3차원 위치·자세(R, t)를 역산하는 표준 기법(`cv::solvePnP`). 보드가 바닥에 놓여 있으면 이 역산으로 "카메라 기준 바닥 평면"이 확정되고, 그 평면에 대한 호모그래피는 **대응점 없이 수학적으로 유도**됩니다. 대형 시트로 영역을 덮을 필요(외삽 문제)가 원리적으로 사라지는 이유입니다.
- **체커보드 1장(A2)의 두 가지 역할**: ① 손에 들고 여러 자세로 촬영하면 내부 파라미터(Step 1), ② 바닥 3지점에 잠깐씩 놓고 찍으면 PnP로 카메라 자세·바닥 평면(Step 2). 별도의 기준점 표식·대형 시트·줄자 실측이 모두 불필요합니다.
- **배치 3원칙**: ① 측정 대상 평면에 **밀착**(막대 등으로 띄워 세우기 금지 — 평면이 어긋나면 그 높이만큼 시차 오차 유입), ② 방향·회전은 무관하되 **들뜸·휨 금지**(출력물은 하드보드/아크릴에 부착), ③ 보드 3지점은 도장 영역에 **넓게 분산**할수록 평면 피팅이 강건해집니다.
- **순서가 고정인 이유**: 호모그래피는 "왜곡이 풀린" 픽셀을 전제로 하므로 내부 파라미터가 먼저여야 하고, 좌표계가 정의돼야 코너 실좌표가 결정됩니다.

### B.2 수행 위치·통신 규격

**수행 장소·주체**: 카메라 최종 설치·고정이 끝난 현장에서, 서버 프로그램의 "캘리브레이션 모드"로 수행합니다. 카메라를 옮기거나 렌즈 설정을 바꾸면 해당 단계부터 재수행합니다.

**연산 위치**: 모든 검출·계산·저장은 **서버 RPi에서 수행**합니다(RTSP 영상과 OpenCV가 서버에 있음). Windows 노트북(Qt)은 버튼 조작과 결과 확인 **UI 역할만** 합니다. `camera_intrinsics.yaml`, `H_floor.yaml`, `H_marker.yaml`은 서버에만 존재합니다.

**캘리브레이션 중 데이터·통신 규격**:

| 구간 | 규격/형식 | 흐르는 데이터 |
|---|---|---|
| 카메라 → 서버 | RTSP/RTP (H.264) | 실시간 영상 (검출·계산은 전부 서버) |
| Qt → 서버 | TCP/TLS + JSON | CALIB_* 명령 (부록 C.6) |
| 서버 → Qt | TCP/TLS + JSON/JPEG | 검출 오버레이 미리보기, 잔차 리포트 |
| 서버 로컬 디스크 | YAML 파일 | K·dist, H_floor, H_marker |
| 서버 ↔ 로봇 | 사용 안 함 | 캘리브레이션 중 로봇 정치는 수동(전원만 ON, 구동 명령 없음) |

**준비물**:
- **체커보드 1장 (전 단계 공용)**: A2 권장(A3 가능), 격자 25~40mm, 하드보드/아크릴에 평평하게 부착. **보드 두께를 실측해 기입**(Step 2에서 법선 방향으로 차감).
- 마커 부착 완료 로봇(**ArUco 중심을 펜 축(회전 중심)과 동축 장착 권장**. 상판 20×17cm 기준 ArUco 12~15cm로 최대 크기 장착). **마커 높이 h**(바닥→마커 면 수직 거리, h≈15cm)를 자로 1~2mm 정밀 실측해 기입.
- (선택) 상시 기준용 소형 ArUco 4개.
- 대형 바닥 시트는 **불필요**. PnP 방식(Step 2~3)이 기본이며, Step 5 외삽 검증이 기준 미달일 때만 대응점 병합 방식을 대안으로 고려합니다.

### B.3 캘리브레이션 시나리오 (Phase A — 설치 후 1회, 순서 고정)

**Step 1 — 카메라 내부 파라미터 (K, dist)**
- 체커보드는 **1장만 사용**합니다. "15~20장"은 보드 개수가 아니라 **촬영 횟수**입니다 — 사람이 보드를 들고 카메라 앞에서 위치·거리·기울기를 바꿔가며 15~20회 촬영합니다. 바닥에 여러 장을 깔아두는 것이 아닙니다.
- 조작 흐름: 작업자가 보드를 든 상태에서 Qt [캡처] 클릭 → `CALIB_CAPTURE` 송신 → 서버가 최신 프레임에서 `findChessboardCorners` 수행 → 성공 여부 + 코너 오버레이 JPEG 회신 → Qt에 표시. 화면 중앙과 네 모서리를 골고루 커버할 때까지 반복.
- Qt [계산] → 서버 `cv::calibrateCamera` → RMS 재투영 오차 회신(합격 기준: 0.5px 미만) → Qt [저장 승인] → 서버가 `camera_intrinsics.yaml` 저장.
- 이후 모든 좌표 계산은 왜곡 보정 후의 픽셀을 사용합니다.
- 재수행 조건: 렌즈 줌·포커스·해상도 프로파일 변경 시. PNO-A9081RG는 전동 바리포컬이므로 **캘리브레이션 후 줌/포커스 조작 금지**를 운영 규칙으로 명시합니다.

**Step 2 — 카메라 외부 파라미터·바닥 평면 확정 (소형 보드 PnP, 3지점)**
- 대형 시트가 필요 없습니다. Step 1과 같은 **A2 보드 1장**을 도장 영역의 **중앙과 대각 양끝 등 3곳 이상**에 잠깐씩(지점당 몇 초) 바닥에 밀착해 놓고, 지점마다 Qt [캡처] 1회.
- 서버: 각 지점에서 코너 서브픽셀 검출 → `cv::solvePnP`(K·dist 사용)로 보드의 3차원 자세(R, t)를 역산 → "카메라 기준 바닥 평면" 후보를 지점 수만큼 확보.
- 3개 이상의 평면 후보를 **하나의 바닥 평면으로 강건 피팅**합니다. 평면 간 불일치량(각도·거리)을 함께 산출하여 Qt에 표시 — 이 값이 곧 시야 전체 외삽 오차의 추정치입니다. 보드 두께는 법선 방향으로 일괄 차감.
- **작업장 좌표계 = 1번 지점 보드의 지정 코너(원점)와 긴 변(X축)**을 피팅 평면에 투영하여 정의. 줄자 실측 불필요.

**Step 3 — H_floor·H_marker 해석적 유도 (계산만, 현장 작업 없음)**
- H_floor: 확정된 K·R·t와 바닥 평면으로부터 평면 유도 호모그래피(`H = K·[r1 r2 t]` 형태)를 **계산으로 직접 유도** → 잔차 리포트 확인 → `H_floor.yaml` 저장. 대응점 스프레드에 의존하지 않으므로 시야 전체에서 유효하며, 정확도는 Step 1 내부 파라미터(특히 왜곡 모델) 품질에 좌우됩니다 — **Step 1을 대충 하면 안 되는 이유**.
- H_marker: 바닥 평면을 **법선 방향으로 실측 마커 높이 h만큼 평행 이동**한 평면에서 동일하게 유도 → `H_marker.yaml` 저장. h 실측 오차의 기여는 δh × (수평거리/카메라높이) 수준으로, δh=2mm면 약 1~2mm입니다.
- 이 방식에서는 "로봇을 코너 4~8곳에 정치"하는 절차가 사라지고 **Step 4의 검증으로 대체**됩니다.

**Step 4 — 로봇 스팟 체크 (마커-펜 사슬 검증)**
- 보드를 1번 지점에 놓은 상태에서 로봇 펜 끝을 기지 코너에 **2~3회 정렬** → Qt [포인트 기록] → H_marker 변환값과 코너 실좌표 비교. 마커-펜 동축 장착, h 실측, 평면 피팅이 전부 이 한 번에 검증됩니다.
- 오차 패턴으로 원인 구분: 한쪽으로 일정하게 밀림 = 마커-펜 오프셋 또는 h 오차 / 위치마다 다르게 튐 = 평면 피팅 또는 왜곡 보정 문제.
- **금지 사항 1**: 오도메트리(스텝 카운트) 주행 결과를 기준 좌표로 사용 금지(순환 참조).
- **금지 사항 2**: 막대 끝에 마커/기준점을 만들어 세우는 방식 금지 — 호모그래피는 하나의 평면에서만 성립하므로 막대 높이만큼 시차 오차가 그대로 유입됩니다.

**Step 5 — 정적 검증 (외삽 확인 포함)**
- 보드를 캘리브레이션에 사용하지 않은 **원거리 지점(도장 영역 가장자리)**에 놓고, 검출 코너 좌표와 H_floor 예측의 잔차를 확인합니다 — 대형 시트 없이 외삽 오차를 정량 확인하는 절차입니다.
- 합격 기준: 중앙 존 잔차 3mm 이하(SRN-004). 불합격 시 Step 1(내부 파라미터)부터 점검 후 Step 2 재수행. 반복 불합격 시 대안으로 보드 다지점 대응쌍 병합 방식을 적용합니다.

**Step 6 — 동적 검증 (검증 전용)**
- "2m 직진" 명령 후 비전 측정 거리와 스텝 카운트 환산 거리를 비교.
- 산출: 호모그래피 교차 검증 + 스텝-거리 스케일(바퀴 지름 보정값). 이 결과를 호모그래피 재계산에 역사용하지 않습니다.

**Step 7 — (선택) 상시 기준 마커 설치**
- 도장 영역 밖 4모서리에 소형 ArUco를 부착하고, 현재 H_floor로 각 마커의 좌표를 역산·기록합니다.
- 용도: 매 세션 시작 시 자동 비교로 카메라 미세 이동을 감지하고, 이동 시 이 4점만으로 호모그래피를 자동 재산출.
- 생략 가능 조건: 카메라 견고 고정 + "카메라 접촉 시 Step 2부터 재수행" 규칙 준수. 외부인 접촉 가능성이 있는 데모 환경에서는 설치 권장(출력물 부착 수준의 비용).

**산출물**: `camera_intrinsics.yaml`, `H_floor.yaml`, `H_marker.yaml` — 서버 기동 시 로드하며, 파일 부재/버전 불일치 시 작업 개시를 차단합니다.

### B.4 런타임 적용 위치 (Phase B — 서버 RPi)

**Step 0 — 세션 시작 체크 (Step 7의 상시 기준 마커 설치 시)**
- 상시 기준 ArUco의 검출 좌표를 저장값과 비교. 임계 초과 시 "재캘리브레이션 필요" 경고 또는 4점 자동 재산출(SR-VIS-003). 미설치 시 "카메라 접촉 금지" 운영 규칙으로 대체.

**경로 ① 로봇 측위 (베이스라인: 온카메라 검출)**
- 카메라 앱이 ArUco 코너 px + 타임스탬프를 서버로 송신(IF-TCP-003) → 서버가 `cv::undistortPoints` → **H_marker** 변환 → 펜 끝 (x, y)·헤딩 θ → 이상치 제거·평활화 → **POSE로 로봇에 중계(10~15Hz)** → 로봇 경로 추종 루프 입력(SR-MPU-005).
- 폴백: 카메라 앱 장애 시 서버가 서브 스트림(최신 1장 유지)에서 자체 검출 — 이때도 코너 점만 undistortPoints로 보정(전체 프레임 remap은 고비용).

**경로 ② 사용자 경로 좌표화 (작업 시작 시 1회)**
- Qt 작도 픽셀 → undistortPoints → **H_floor** 변환 → 작업장 좌표 → 획 분해 + 리드인/리드아웃 부가 → 유도 루프의 목표 경로.
- 변환 수행 위치: Qt(Windows 노트북)는 화면 픽셀 좌표만 서버로 송신하고, undistortPoints·H_floor 변환·획 분해는 모두 **서버**가 수행합니다(H 파일이 서버에만 존재).

**경로 ③ 결과 검사 (작업 완료 후)**
- 메인 스트림 고해상도 캡처 → 도포 선 색 기반 검출 → **H_floor** 변환 → 계획 경로 대비 평균/최대 이탈(mm)·누락 구간 리포트.

**혼용 금지 재확인**: ①은 반드시 H_marker, ②③은 반드시 H_floor. 소스코드에서 `toWorldFromMarker()` / `toWorldFromFloor()`처럼 함수 인터페이스를 분리하여 오용을 구조적으로 차단합니다.

## 부록 C. 서버 ↔ 로봇 메시지 규격 (ICD 초안)

### C.1 공통 규칙

- 전송 계층: IF-WIFI-001의 TCP/TLS **단일 상시 세션**. 로봇 RPi가 클라이언트로서 서버에 접속·유지하며, 끊기면 자동 재접속을 시도합니다(재접속 중에는 로봇 자체 워치독으로 정지 상태 유지).
- 인코딩: 개발 단계에서는 JSON(디버깅·로깅 용이), 통합·최적화 단계에서 고정 길이 바이너리로 전환 가능. 필드 의미는 동일하게 유지합니다.
- 모든 메시지 공통 필드: `type`(메시지 종류), `seq`(단조 증가 일련번호 — 유실·순서 뒤바뀜 감지), `t`(송신 시각 ms).

### C.2 하향 메시지 (서버 → 로봇 RPi)

| 메시지 | 주기 | 주요 필드 | 설명 |
|---|---|---|---|
| PATH | 작업 시작 시 1회 | 획 배열[웨이포인트(x,y)…, 리드인/아웃 구간, 노즐 ON/OFF 위치, 속도 계획] | 서버가 계산한 최종 경로. 로봇이 저장 후 추종 |
| POSE | 10~15Hz | x, y, θ (펜 끝), t_frame (카메라 프레임 시각), conf | 로봇 경로 추종 루프의 절대 측위 입력. **상시 공급 필수** |
| CMD_NOZZLE | 이벤트 | on (0/1) | 수동 오버라이드 전용(기본 노즐 판정은 로봇이 PATH 기반 위치 통과로 자체 수행) |
| CMD_ESTOP | 이벤트 | reason | 비상 정지 (장애물·측위 상실·운영자 조작·통신 이상) |
| CMD_MODE | 이벤트 | mode | IDLE / RUN / CALIB 상태 전환 |

### C.3 상향 메시지 (로봇 RPi → 서버)

| 메시지 | 주기 | 주요 필드 | 설명 |
|---|---|---|---|
| STATUS | 10Hz | left_steps, right_steps (누적 uint32), state, stall_flag, batt_mV, obstacle_mm, nozzle_state | 서버 측 지연 보상(SR-VIS-005)·탈조 교차 검증(SR-STM-005)·Qt 대시보드 표시의 입력 |
| EVENT | 즉시 | code, detail | ESTOP 발생, 장애물 감지, UART 이상, 저전압 경고 등 |
| ACK | 이벤트 | ack_seq | CMD_ESTOP 등 중요 명령의 수신 확인 |

### C.4 워치독 (SRN-001의 구체화)

- 로봇 측: 유효한 **POSE를 300ms** 이상 미수신(또는 신뢰도 0 연속) → 자체 ESTOP(감속 램프 정지) + 노즐 강제 차단 + EVENT 보고.
- 서버 측: STATUS를 **500ms** 이상 미수신 → 작업 일시 정지 판정 + Qt 대시보드 경고 표시. 측위 피드(IF-TCP-003) 두절 시 서브 스트림 폴백 검출로 전환, 전환 실패 시 정지 지시.

### C.5 로봇 내부 UART 프레임 (RPi ↔ STM32)

프레임 구조: `[STX 0xAA][LEN][CMD][PAYLOAD ...][CRC8][ETX 0x55]`, 115200bps 8N1.

| CMD | 방향 | PAYLOAD | 설명 |
|---|---|---|---|
| 0x01 SET_SPEED | RPi→STM32 | int16 left_sps, int16 right_sps | 부호 = 회전 방향. 발행 상한 20~50Hz |
| 0x02 NOZZLE | RPi→STM32 | uint8 on | 도장 서보 PWM 제어 (SR-STM-002) |
| 0x03 ESTOP | RPi→STM32 | uint8 reason | 최우선 처리, 감속 램프 정지 + 노즐 차단 |
| 0x81 STATUS | STM32→RPi | uint32 left_steps, uint32 right_steps, uint8 flags | 10Hz 주기 보고 (누적 스텝 카운트) |

- CRC 불일치 프레임은 폐기하고 다음 프레임을 대기합니다(재전송 요구 없음 — 다음 주기 명령이 곧 도착하므로).
- STM32는 SET_SPEED를 300ms 이상 미수신 시 자체 감속 정지합니다(**이중 워치독**: Wi-Fi 단계와 UART 단계 각각 독립 동작).

### C.6 서버 ↔ Qt 캘리브레이션 메시지 (IF-NET-002, TCP/TLS + JSON)

| 메시지 | 방향 | 주요 필드 | 설명 |
|---|---|---|---|
| CALIB_MODE | Qt→서버 | on (0/1) | 캘리브레이션 모드 진입/종료 (진입 시 로봇 RUN 금지) |
| CALIB_CAPTURE | Qt→서버 | - | 최신 프레임에서 체커보드 코너 검출 시도 (Step 1~3) |
| CALIB_POINT | Qt→서버 | corner_index | 로봇 정치 포인트 기록 — 서버가 ArUco 중심 검출 후 해당 코너 좌표와 대응 저장 (Step 4) |
| CALIB_COMPUTE | Qt→서버 | target (intrinsic / floor / marker) | 해당 대상 계산 실행 |
| CALIB_RESULT | 서버→Qt | ok, rms, max_err, n_points, preview_jpeg | 계산 결과·잔차·검출 오버레이 미리보기 |
| CALIB_SAVE | Qt→서버 | target | 승인 → 서버가 YAML 저장 |
| CONFIG_SYNC | 서버→Qt/로봇 | H_floor 사본, K·dist, 규격 버전 | 로그인/접속 시 설정 배포. Qt는 Top-view 표시용으로만 사용(제어용 변환의 정본은 서버) |

## 부록 D. 용어집 (Glossary)

| 용어 | 설명 |
|---|---|
| 내부 파라미터 (K, dist) | 렌즈의 초점거리·주점·왜곡계수. 체커보드 1장을 15~20회 촬영하여 1회 산출. 렌즈 설정 변경 시에만 재수행 |
| 호모그래피 (H) | 평면↔평면을 잇는 3×3 변환 행렬. 자유도 8 → 점 대응 최소 4쌍 필요. 본 시스템은 H_floor(바닥)·H_marker(마커 높이) 2개 사용 |
| 기준점 | 실좌표를 정확히 아는 평면 위의 점. 본 시스템에서는 체커보드 격자 코너들이 기준점 역할(별도 표식 상시 부착 불필요) |
| PnP | Perspective-n-Point. 크기를 아는 패턴의 영상 투영으로부터 카메라-패턴 간 3차원 자세(R, t)를 역산하는 기법(`cv::solvePnP`). 바닥 평면 확정 → 호모그래피 해석적 유도의 기반 |
| 외삽/보간 | 보정점들이 감싸는 영역 안은 보간(정확), 밖은 외삽(오차 증가). 보정점이 도장 영역을 덮어야 하는 이유 |
| 서브픽셀 검출 | 코너 위치를 픽셀 격자보다 세밀한 소수점 좌표로 추정하는 기법. 측위 정밀도의 기반 |
| 메인/서브 스트림 | 한 카메라가 서로 다른 인코딩 설정으로 동시 송출하는 2개의 영상. 프로파일마다 RTSP URL 별도 존재 (2.3절) |
| GOP | I-프레임(독립 복호 프레임) 사이의 간격. 짧을수록 저지연·빠른 복구 |
| B-프레임 | 미래 프레임을 참조하는 압축 프레임. 지연을 유발하므로 측위 스트림에서 비활성 |
| 3D-DNR | 여러 프레임을 시간축 평균하는 카메라 노이즈 제거 기능. 이동체에 잔상 유발 → 측위 스트림에서 OFF (카메라 웹 설정, 코드 구현 아님) |
| sps | steps per second. 스텝모터 속도 단위 (스텝 주파수) |
| 탈조 (step loss) | 스텝모터가 과부하·급가감속으로 명령 펄스를 놓쳐 실제 회전이 명령보다 적어지는 현상 |
| 리드인/리드아웃 | 획 시작 전/종료 후에 부가하는 노즐 OFF 가상 주행 구간. 오차 수렴용 |
| 워치독 | 일정 시간 신호 부재 시 자동으로 안전 동작(정지·차단)을 수행하는 감시 메커니즘 |
| ICD | Interface Control Document. 모듈 간 통신 규격 정의 문서 (부록 C) |
| SUNAPI | 한화비전 카메라의 HTTP 기반 설정·제어 API. 프로파일 설정, 스냅샷, 이벤트 등을 코드로 제어 |
| Wisenet Open Platform | 한화비전 카메라에 커스텀 앱(.cap)을 설치·실행하는 플랫폼. SDK는 STEP 포털에서 제공 (2.5절 확장 옵션의 기반) |
| WiseAI | 카메라 내장 AI(사람·차량 등 고정 객체 검출·속성). ArUco 같은 커스텀 패턴에는 직접 사용 불가 |

## 부록 E. 파트별 역할 정의 및 운용 시나리오

### E.1 파트별 역할 정의 (확정)

**공유기 — "전송만, 연산 없음"**
- 유선 연결: CCTV, 서버 RPi, (권장) Qt 노트북 / 무선 연결: 로봇 RPi.
- 모든 노드 고정 IP(DHCP 예약). 데모 환경은 전용 공유기 사용.

**CCTV (한화비전 PNO-A9081RG) — "로봇의 눈: 온카메라 ArUco 검출 (베이스라인)"**
- Open Platform 앱(.cap)이 원시 프레임에서 ArUco를 검출하고 **코너 px + 타임스탬프 + 신뢰도**를 서버에 15~30Hz 송신(IF-TCP-003, SR-CAM-001).
- RTSP 메인 스트림(관제·검사) 상시 송출 + 서브 스트림(서버 폴백 검출용) 유지.
- 웹 설정 기반 1차 화질 전처리(3D-DNR·B-프레임 OFF, GOP 단축), SUNAPI로 설정·앱 상태 확인(IF-HTTP-002).
- H 계산·캘리브레이션은 하지 않음(서버 담당). WiseAI는 고정 객체용이라 ArUco에 직접 사용 불가(커스텀 앱 필요).

**Qt 클라이언트 (Windows 노트북) — "표시와 조작"**
- 캘리브레이션 버튼 조작: 명령은 **서버로** 송신합니다(CALIB_*, 부록 C.6). CCTV로 직접 명령을 보내지 않습니다.
- 관제 표시: 영상 표시 + 서버가 배포(CONFIG_SYNC)한 H_floor·K 사본으로 **Top-view 표시용 워핑을 로컬 수행**(노트북 CPU 활용). 표시 전용이며, 제어에 쓰이는 좌표 변환의 정본은 항상 서버입니다.
- 경로 작도: 화면 픽셀 좌표 시퀀스만 서버로 송신(변환·획 분해는 서버).
- 대시보드·결과 리포트 표시, 사용자 로그인(ID/PW, SRN-006).

**서버 RPi — "좌표 변환·경로 계산·중계의 주체"**
- 캘리브레이션 전담: 내부 파라미터·PnP·H_floor·H_marker 계산 및 yaml 저장(부록 B).
- 측위 중계: 카메라 앱의 코너 px 수신 → undistortPoints → H_marker → 펜 끝 POSE 생성 → **로봇에 10~15Hz 중계**(SR-VIS-004). 폴백 시 서브 스트림 자체 검출.
- 경로 계산: Qt 작도 픽셀 → H_floor 변환 → 획 분해 → 리드인/아웃·노즐 위치·속도 계획 포함한 **PATH를 로봇에 1회 전송**.
- 결과 검사·정량 리포트 생성, 설정 배포(CONFIG_SYNC), 사용자 계정·이력 관리(SRN-006).
- 고주기 유도 제어는 수행하지 않음(로봇 RPi 담당) — 서버 부하 최소화.

**로봇 RPi — "경로 추종과 안전의 주체"**
- PATH 수신·저장, POSE(10~15Hz) 수신 → 지연 보상 → 횡방향/헤딩 오차 → (v, ω) 생성 → 좌/우 sps 변환 → UART로 STM32 전달(SR-MPU-005). STATUS 10Hz 상향.
- 노즐 ON/OFF를 PATH 기반 위치 통과로 자체 판정(기구 지연 선행 보정).
- 워치독(POSE 300ms 미수신 시 정지+노즐 차단), 커널 GPIO/PWM 드라이버(타워램프·부저), 초음파/ToF 커널 ISR 긴급 정지.

**STM32 — "펄스와 즉응"**
- 좌/우 스텝 펄스 생성(사다리꼴 가감속), 누적 스텝 카운트 10Hz 보고, 노즐 서보 PWM, UART 워치독(이중), 긴급 정지 최우선 처리.

### E.2 운용 시나리오 (전원 ON → 작업 완료)

**S0 — 기동·로그인**
- 각 장치 전원 인가 → 서버가 yaml 3종 로드(부재 시 "캘리브레이션 필요" 상태로 작업 차단) → 로봇 RPi가 서버에 TLS 접속·STATUS 송신 개시 → Qt 로그인(ID/PW) → 서버가 CONFIG_SYNC로 해당 사용자의 설정·H 사본 배포 → (Step 7 상시 마커 설치 시) 카메라 이동 자동 체크.

**S1 — 캘리브레이션 (필요 시에만, 부록 B)**
- 최초 설치, 카메라 접촉/이동, 렌즈 설정 변경 시에만 수행. Qt 버튼 ↔ 서버 계산 ↔ yaml 저장의 반복(Step 1~5).

**S2 — 경로 작도·승인·PATH 전송**
- 사용자가 Qt의 Top-view 위에 직선/사각형/주차 구획선 작도 → 픽셀 시퀀스를 서버로 송신 → 서버가 H_floor 변환·획 분해·리드인/리드아웃·노즐 위치·속도 계획 부가 → 계획 경로와 로봇 초기 배치 안내 위치를 Qt에 오버레이 회신 → 사용자 [작업 시작] 승인 → 서버가 **PATH를 로봇에 1회 전송** + RUN 모드 전환(CMD_MODE).

**S3 — 자율 도장 실행 (폐루프)**
- 카메라 앱: ArUco 코너 px + 타임스탬프 검출·송신(15~30Hz) → 서버: H_marker 변환 → 펜 끝 POSE → **로봇에 중계(10~15Hz)**.
- 로봇 RPi: POSE 지연 보상 → 횡방향/헤딩 오차 → (v, ω) → 좌/우 sps → UART SET_SPEED → STM32 스텝 구동. 노즐은 PATH 기반 위치 통과 자체 판정.
- 로봇→서버 STATUS(10Hz) → 서버→Qt 실시간 위치·궤적·상태(속도, 배터리, 탈조 경고) 표시. 서버는 POSE와 스텝 카운트 추정치 괴리를 감시(탈조 판정, SR-STM-005).
- 이상 시: 장애물(커널 ISR)·POSE 유실(워치독)·측위 상실(신뢰도 0 연속) → 정지 + 노즐 차단 + EVENT 보고. 카메라 앱 장애 시 서버가 서브 스트림 폴백 검출로 전환.

**S4 — 완료·검사**
- 마지막 획 종료 → 서버가 메인 스트림 고해상도 캡처 → 도포 선 검출 → H_floor 좌표화 → 평균/최대 이탈(mm)·누락 구간 리포트 → Qt 표시 및 사용자 이력 저장.

**S5 — 종료**
- 로봇 IDLE 복귀, 세션 로그(명령·측위·이벤트) 서버 저장.
